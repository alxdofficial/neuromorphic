# V2.1 — Representation Learning (Problem 1)

Working design doc for the representation-learning module of V2.1. Authored
2026-05-19 on the `v2.1` branch off the `v2-final` tag.

Implementation: `src/repr_learning/` and `scripts/repr_learning/`. All three
variants (V2.1 + two baselines) verified end-to-end with the forward smoke
test on Llama-3.2-1B.

## Context: two-problems framing

V1.5, V2, and flat-bank all tried to solve two distinct ML problems with one
mechanism, which is why none of them cleanly worked:

1. **Representation learning**: "What is the compressed internal language the
   LM should use to encode its own concepts?" — slow plasticity, trained
   end-to-end, the output is a learned vocabulary and grammar.

2. **Session memory**: "Given any language (English, LM hiddens, or our
   learned vocabulary from problem 1), how do we store session-specific
   content within a bounded compute/VRAM footprint?" — fast plasticity,
   inference-time updates.

V2.1 attacks these sequentially. This doc covers **problem 1**: build a
learned vocabulary + grammar that compresses text into a structured graph.
The graph format will be the substrate the memory module (problem 2) later
operates on.

## Thesis statement

A language model can learn an internal compositional language — N=4096
discrete concept codes connected by typed directional edges — that
encodes the content of a 256-token English paragraph in 32 edges. The
graph is learned by reconstruction: encoder maps text → graph, decoder
maps graph → text, both trained end-to-end such that per-token
cross-entropy on the reconstructed text drops to a level competitive
with what the LM would achieve seeing the original text directly.

## Architecture overview

```
text window (~256 tokens)
    │
    ▼
[Encoder: bidirectional transformer]
    │
    ├─→ 32 slot queries (DETR-style learned)
    │
    ▼
[Per-slot prediction heads]
    │  src_q       dst_q        edge_vec
    │  (D=1024)    (D=1024)     (D=128)
    │
    ├─→ Classify src_q against codebook → src_id ∈ {0..4095}
    ├─→ Classify dst_q against codebook → dst_id ∈ {0..4095}
    │   (Gumbel-softmax STE; not VQ-VAE)
    │
    ▼
Memory: 32 edges = {(src_id_k, dst_id_k, edge_vec_k)} for k=0..31
    │
    ├─→ proj_src(codebook[src_id])  → [32, d_llama]
    ├─→ proj_dst(codebook[dst_id])  → [32, d_llama]
    ├─→ proj_edge(edge_vec)         → [32, d_llama]
    │
    │   interleaved: [src_0, edge_0, dst_0, src_1, edge_1, dst_1, ...]
    │
    ▼
[Frozen Llama-3.2-1B decoder]
    │
    │  input: [memory_96, MASKED_text_256]
    │  prepended memory + span-masked text
    │
    ▼
Logits over 256 text positions
    │
    ▼
Loss: cross-entropy on MASKED positions only + VQ commitment loss
```

## Component specifications

### Codebook (concept_id table)

```
concept_id: nn.Parameter[N=4096, D_concept=1024]
```

- 4096 abstract concept anchors
- Each is a 1024-dim learned vector (the "slow weights" — vocabulary)
- Initialized with small Gaussian (std=1/√D_concept ≈ 0.031)
- Updated **only by backprop**; never mutated at inference

We use **classification-style selection**, NOT VQ-VAE. Encoder produces a
1024-dim query per slot; dot-product against codebook gives 4096 scores
per slot; argmax forward, Gumbel-softmax STE backward. Gradient flows
through the chosen embedding via a soft one-hot. Load-balance auxiliary
loss prevents collapse (no commitment loss needed).

### Encoder

```
SmallBiTransformer:
  - Input: Llama token embeddings of the 256-token window
  - Layers: 2 transformer layers, bidirectional self-attention
  - Hidden dim: D_enc = 256
  - Heads: 4
  - FFN dim: 1024
  - Output: [T, 256] contextualized representations per token
```

Slot mechanism (DETR/Set Transformer style):

```
edge_queries: nn.Parameter[32, D_enc=256]   # learned, treated as a set
edge_slots = CrossAttention(
    query = edge_queries,         # [32, 256]
    keys  = text_h,               # [T, 256]
    values = text_h,
)                                  # [32, 256]
```

The slot queries can interact via self-attention (Set Transformer) to
avoid collapsing to the same code. For v0, simple cross-attention with
no inter-slot interaction is sufficient.

### Per-slot prediction heads

```
src_q  = src_head(edge_slots)      # [32, D_concept=1024]
dst_q  = dst_head(edge_slots)      # [32, D_concept=1024]
edge_vec = edge_head(edge_slots)   # [32, D_edge=128]

# Each head is a 2-layer MLP: 512 → 512 → (1024 or 128) (gelu)
```

Classification-style selection (Gumbel-softmax STE):

```python
def gumbel_argmax_ste(scores, temperature=0.5):
    # Gumbel noise: g = -log(-log(U))
    u = torch.rand_like(scores).clamp(1e-9, 1-1e-9)
    gumbel = -torch.log((-torch.log(u)).clamp_min(1e-9))
    noisy = (scores + gumbel) / temperature
    # Forward: hard argmax
    idx = noisy.argmax(-1)
    # Backward: through softmax (one-hot + soft − soft.detach())
    soft = F.softmax(noisy, dim=-1)
    one_hot = F.one_hot(idx, scores.shape[-1]).float()
    one_hot_ste = one_hot + soft - soft.detach()
    return idx, one_hot_ste

# Lookup chosen concept embedding (differentiable):
emb = one_hot_ste @ concept_id   # [32, D_concept]
```

### Memory tokens for Llama

Three projection MLPs map codebook/edge dims to Llama's d_model. Each
projection ends with a LayerNorm so memory token magnitudes stay in
Llama's expected input range (RMS ~ 1.0).

```
proj_src  = Sequential(Linear(1024→1024), GELU, Linear(1024→2048), LayerNorm(2048))
proj_dst  = Sequential(Linear(1024→1024), GELU, Linear(1024→2048), LayerNorm(2048))
proj_edge = Sequential(Linear(128→1024),  GELU, Linear(1024→2048), LayerNorm(2048))
```

Per edge, produce three tokens:

```
src_tok  = proj_src(concept_id[src_id])    # [32, 2048]
dst_tok  = proj_dst(concept_id[dst_id])    # [32, 2048]
edge_tok = proj_edge(edge_vec)             # [32, 2048]
```

Interleaved as memory:

```
memory = interleave([src_tok, edge_tok, dst_tok]).reshape(96, 2048)
# Order: [src_0, edge_0, dst_0, src_1, edge_1, dst_1, ..., src_31, edge_31, dst_31]
```

### Decoder integration

Llama-3.2-1B is **frozen**. Memory tokens are prepended; text tokens
follow:

```python
input_embed = llama.embed(masked_input_ids)        # [256, 2048]
all_embed = torch.cat([memory, input_embed], dim=0)  # [352, 2048]

# Gradient flows through Llama to reach trainable modules
# Llama's own parameters are NOT updated
for p in llama.parameters():
    p.requires_grad = False

logits = llama(all_embed)                          # [352, vocab]
text_logits = logits[96:]                          # [256, vocab]
```

### Loss formulation

**Reconstruction loss**: span-masked text recovery.

```python
# Per training sample, vary the masking difficulty:
mask_ratio ~ Uniform(0.5, 0.9)
span_lengths ~ Uniform(5, 15)  # contiguous spans

mask_positions = sample_span_masks(seq_len=256, target_ratio=mask_ratio,
                                    span_len_range=(5, 15))

masked_input_ids = input_ids.clone()
# At masked positions, replace token embedding with learnable MASK embedding
mask_embed = nn.Parameter(llama.embed_tokens.weight[UNK_ID].clone())
# (used at runtime to substitute embeddings at masked positions)

loss_recon = F.cross_entropy(
    text_logits[mask_positions],
    input_ids[mask_positions],
    reduction='mean',
)
```

Why span masking with high mask ratio:
- Avoids teacher-forcing leak (masked positions can't see their own value)
- 60-80% mask rate forces memory to carry most of the content
- Span structure (vs random per-token) forces multi-token recovery,
  closer to the memory-use pattern
- Mask rate sampled per-batch gives gradient signal at multiple
  difficulty levels

**Load-balance auxiliary loss** (Switch Transformer style; no VQ-VAE
commitment needed because we use classification, not nearest-neighbor):

```python
def load_balance_loss(scores):
    """Penalize correlation between empirical pick frequencies and
    mean softmax probabilities. Min when uniform across all N nodes.
    """
    N = scores.shape[-1]
    flat = scores.reshape(-1, N)
    picks = flat.argmax(-1)
    f = bincount(picks, minlength=N) / picks.numel()  # empirical freq
    P = softmax(flat, -1).mean(0)                      # mean prob
    return N * (f * P).sum()

loss_aux = (load_balance_loss(src_scores) + load_balance_loss(dst_scores)) / 2
```

**Total loss**:

```python
total_loss = loss_recon + λ_aux * loss_aux
λ_aux = 0.01  # gentle pressure; collapse-prevention only
```

## Training data

- Source: web text corpus (FineWeb-Edu sample, ~1B tokens for v0)
- Window: 128-384 tokens, sampled from coherent paragraphs (no
  document-boundary windows)
- Median window length: ~256 tokens
- Training compute: 1 GPU × ~24 hours for first useful checkpoint

## Bottleneck capacity comparison (the bits question)

Each system produces 96 memory tokens for Llama, but they carry
different amounts of information at the bottleneck. Compared across
two axes:

### Strict information-theoretic budget

| System | Discrete bits per chunk | Continuous floats per chunk |
|---|---:|---:|
| **V2.1** (32 edges) | 32 × (12 + 12) = **768 bits** (src + dst IDs) | 32 × 128 = **4096 floats** (edge_vec) |
| **Baseline A** (96 flat codes) | 96 × 12 = **1152 bits** | 0 |
| **Baseline B** (96 × D_cont=48) | 0 | 96 × 48 = **4608 floats** |

### Effective bits under training (rough estimate at ~2-4 bits/float)

| System | Discrete bits | Continuous effective bits | **Total** |
|---|---:|---:|---:|
| V2.1 | 768 | 4096 × 2 = 8,192 | **~9,000** |
| Baseline A | 1,152 | 0 | **~1,150** |
| Baseline B | 0 | 4608 × 2 = 9,216 | **~9,200** |

### Reading the comparison

- **V2.1 vs Baseline B**: ~equal total effective bits. Both ~9K. The
  difference is the *structure* of those bits: V2.1 splits them
  ~10% discrete (where) / ~90% continuous (what kind), connected
  in 32 triples. B has zero structure — 4608 continuous floats over 96
  independent tokens. If V2.1 wins at equal total bits, structure
  matters.

- **V2.1 vs Baseline A**: V2.1 has ~8× more effective bits. A is
  intentionally low-capacity — it's the "discrete only, no continuous
  side-channel" ablation. If V2.1 wins, we've shown that *both*
  discrete and continuous channels are needed; if A actually wins,
  the continuous channel is wasted. Useful as a rate-distortion floor
  point.

### Optional: Baseline A' for fully matched bit budget

If we want a baseline at V2.1's exact bit budget but without the graph
structure, **Baseline A'** = `96 codes (id ∈ {0..4095}) + 96 × d_side
continuous side-channel`. Picking `d_side ≈ 43` matches V2.1's
continuous-float budget (96 × 43 ≈ 4128 ≈ 4096).

A' is not implemented in v0 — we start with A (pure discrete) and B
(pure continuous) as the corner cases. A' can be added if either A or
B is unfair to V2.1's specific design.

### What "compression" means here

The original text window is ~256 tokens × log₂(128,256 vocab) ≈ 4,400
bits. At ~9K effective bits in the bottleneck, all three systems have
roughly 2× *more* capacity than the input text — so the bottleneck
isn't compressing in the bits sense. The TEST is whether the
*structure* of that capacity supports reconstruction:

- Random encoder + 9K bits: would NOT reconstruct (capacity wasted)
- V2.1 / Baseline B / A: trained encoders that learn to USE the
  capacity. The reconstruction CE measures how usefully they use it.

This is a "discovery-of-structure" task more than a "compression"
task. The interesting question is whether V2.1's graph-structured
allocation of capacity beats flat allocations of similar size.

## Evaluation plan

### Primary metric

**Per-token CE on masked positions** at held-out test windows. Compare:

| Baseline | Expected CE | What it tests |
|---|---:|---|
| Random token guessing | log(vocab) ≈ 11.4 | upper bound |
| Llama no context (asked to fill masks blind) | ~5-8 nats | floor |
| Llama with original text (teacher-forced)* | ~1-2 nats | ceiling |
| **Ours (32 edges + frozen Llama)** | **TBD** | our system |

*The teacher-forced ceiling is informational only — it's what Llama would
get if memory were lossless. We expect to fall somewhere between the
floor and ceiling. The closer to the ceiling, the better the memory is.

### Secondary metrics

- **Codebook utilization**: % of codes used at least once on held-out
  data. Target: > 70%. Below 30% suggests codebook collapse.
- **Mask-rate scaling**: CE as a function of mask ratio (0.3, 0.5, 0.7, 0.9).
  Steep CE growth at high masking = memory not carrying enough content.
- **Compression ratio at fixed CE quality**: at the elbow of the
  rate-distortion curve, how many tokens of original text are encoded per
  byte of memory? Should be > 1.0 (compression positive).
- **Code interpretability**: cluster 4096 codes by their most-frequent
  activating context windows. Do clusters look semantic ("person names,"
  "verbs of motion," "numbers")?
- **BERT cosine on reconstructed text vs original**: lossy upper-bound
  measure — does the model preserve semantic content even when surface
  form drifts?

### Success criteria for v0

We declare the rep-learning module "working" if:

1. Per-token CE at 70% mask rate < 4.0 nats (well below the no-context
   floor; comfortably below random)
2. Codebook utilization > 50%
3. Code clusters show qualitative semantic structure
4. The system scales gracefully — increasing mask rate raises CE
   monotonically but doesn't blow up

If we hit these, the codebook + encoder are good enough to be the
substrate for the memory module (problem 2).

## Connection to memory module (problem 2)

Once rep-learning is working, the trained codebook + encoder become
*frozen inputs* to the memory module:

- The codebook (4096 codes) is the abstract vocabulary
- The encoder takes any text → edges (src_id, dst_id, edge_vec)
- The memory module accumulates edges across multiple sessions/chunks
- Memory's job: maintain a bounded edge store with pruning/merging,
  retrieve relevant edges given a query

The memory module doesn't need to learn the codebook — it's given. This
decouples the two research problems cleanly. We can iterate on memory
mechanisms (Hebbian, pruning policies, retrieval strategies) without
re-training the vocabulary every time.
