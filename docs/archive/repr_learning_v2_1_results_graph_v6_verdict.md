## Verdict

**The compressed memories do not beat a no-memory floor.** The floor (vanilla, no
context — Llama parametric priors only) scores Judge **13.0**; every 274,944-float
compressor — mamba 13.2, continuous 12.5, MT 11.2, graph_v6 **10.9**, flat 10.9 — lands
**at or below** it (differences are ~±a-few/336, i.e. noise). graph_v6 (the primary arm)
is mid/bottom-pack, statistically tied with flat and indistinguishable from the floor.
Meanwhile the **ceiling** (full ~8k context, no compression) scores **38.2** — so the task
is solvable; the **compression bottleneck**, not the architecture, is where it breaks.
~85% of every compressor's answers are `wrong_value` (confident confabulation), echoing the
graph_v5 verdict — but now under a *fair, param-matched* comparison (all arms 274,944-float
state, ~48M memory params within 0.14%, identical rank-16 LoRA, shared sinusoidal PE), which
rules out an implementation handicap as the explanation. graph_v6's relational state is
mechanically *alive* (train `state_effect` ≈ 3.4 vs graph_v5's ≈ 0), but that liveness does
not translate into better downstream QA at this compression ratio. The open question is no
longer "is the graph read dead?" (it isn't) but "why does a live, fairly-matched memory not
beat parametric recall at 61× compression?" — next probes: lower compression (bigger state),
and a read-utilization ablation (does decoding actually depend on the retrieved facts?).
