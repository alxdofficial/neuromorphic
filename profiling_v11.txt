=== Remaining Performance Issue ===

360 tok/s with PyTorch sequential loop (T=128 × R=4 = 512 steps).
Each step does: cell_gather + border_gather + scatter + F.linear × 4 + decay.
With gradient checkpointing, each step is computed 2× (fwd + recompute in bwd).

The bottleneck is NOT the MLPs (F.linear is fast). It's:
1. 512 sequential Python steps with kernel launch overhead per step
2. cell_gather uses advanced indexing: msg[batch_idx, cell_idx, conn_expanded]
   which creates [BS, NC, C, K, D] intermediate = 31K × 16 × 8 × BS × 4 bytes
3. border_gather similarly creates intermediates
4. Gradient checkpointing doubles the forward compute

=== Path to Higher Throughput ===

Option A: Adopt v9-style base/inject decomposition
  - State MLP: base_hidden = W_in@received + W_prim@prim + W_id@id + W_decay@decay (constant across T)
  - inject_hidden = W_in@inject[t] (varies per T, batch all T via one F.linear)
  - hidden = tanh(base_hidden + inject_hidden) for all T at once
  - update = tanh(W2@hidden) for all T at once
  - fused_scan for h = decay*h + (1-decay)*update
  - Then msg MLP batched over T
  - This eliminates the T loop entirely for the MLP. Only the R loop remains.
  - Expected: T×R=512 steps → R=4 batched passes, each with 2 F.linear + 1 scan
  - ~128× fewer kernel launches

Option B: Reduce R from 4 to 1-2
  - R=1: each token only does 1 round (direct neighbors only)
  - R=2: 2 rounds (2-hop propagation)
  - Cuts compute linearly

Option C: Combined
  - v9-style scan for temporal integration + R=2 rounds
  - Expected: ~4 batched passes (2 rounds × 2 passes if doing base/inject split)
  - Each pass: 2 F.linear + gather + 1 scan = ~4 kernel launches
  - Total: ~16 kernel launches instead of 512
  - Expected throughput: 5-15K tok/s depending on gather overhead
